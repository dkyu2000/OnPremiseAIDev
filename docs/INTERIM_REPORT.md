## 2.4 (신규) 선택지 L(Nemotron) 실전검증 — OpenCode 실사용 테스트 결과

승격 후보로 판정된 Nemotron을 격리된 PoC/소크 테스트를 넘어 **실제 OpenCode 개발 워크플로**
(OpenSpec propose→apply, Playwright E2E 테스트까지)에 투입해 실전검증했습니다. 격리 테스트에서는
드러나지 않던 **인프라 버그 4건**을 발견해 전부 조치했고, 최종적으로 **실제 앱(Todo 앱) 구현부터
Playwright 브라우저 자동화 테스트 통과까지 엔드투엔드 완주**를 확인했습니다.

**발견·조치한 버그 4건**

1. **reasoning-parser 미설정** — Nemotron의 사고과정(CoT) 원문이 답변 본문에 그대로 섞여 나오던
   문제. `--reasoning-parser nemotron_v3` 추가로 해결(vLLM 내장 파서, 최초 등록 시 누락돼 있었음).
2. **장시간 세션 컨텍스트 초과** — 에이전트 작업이 길어지면 자동 압축(요약) 요청 자체가 컨텍스트
   한도 초과로 거부되어 작업이 멈추는 현상. 근본 원인은 컨텍스트 한도(65,536)가 Nemotron 자체
   한계가 아니라 다른 모델 설정을 그대로 물려받은 값이었던 것 — Nemotron은 실제로 262,144토큰까지
   지원. **128K(131,072)로 상향**해 여유 마진을 2배 이상 확보.
3. **LiteLLM 게이트웨이 토큰/비용 표시 버그** — OpenCode 화면의 Context/토큰/비용 패널이 항상
   0으로 고정되던 문제. LiteLLM 자체의 알려진 버그 2건(스트리밍 usage 정보 유실)이 겹친 것으로
   확인, 임시 패치로 해결(근본 해결책인 중계 프록시 전환 방안은 설계까지 완료, 필요 시 적용 가능).
4. **presidio-analyzer 컨테이너 순간 재시작** — PII 마스킹 게이트웨이가 짧게 재시작하며 일시적
   500 오류 발생(자동 재시도로 복구됨, 재발 원인은 계속 모니터링 중).

**최종 실사용 검증 결과**: 위 조치 이후, OpenCode로 Flask 기반 Todo 앱을 처음부터 끝까지 구현시키고
**Playwright(Python 라이브러리, 실제 Chromium 브라우저 자동화)로 UI/API 테스트를 작성·실행**해
CRUD 전체(추가/완료토글/삭제/API 4종)가 실제 브라우저 조작으로 통과하는 것까지 확인했습니다 —
"실제 개발 업무에 투입 가능한 수준"임을 실전 시나리오로 입증한 첫 사례입니다.

**남은 과제**: G/S/H(신규) 선택지도 동일한 실전검증(OpenCode 비교 + 부하테스트)이 필요하며, 발견된
4건 중 reasoning-parser 누락 같은 유형은 다른 모델 전환 시에도 재발할 수 있어 향후 신규 모델 등록
시 체크리스트에 반영할 예정입니다.
