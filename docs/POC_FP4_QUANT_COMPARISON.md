# PoC 결과 — FP4 양자화 vs 모델 크기 품질·속도 비교

> 목적: 운영 모델 양자화 선택(특히 "70B-NVFP4 vs 작은 FP8") 의사결정 근거 확보.
> 검증장비: RTX 5090 32GB (Blackwell SM120). 측정일: 2026-06-30.
> 재현: `scripts/poc_quant_compare.py <엔드포인트> <이름> [easy|hard]` — 각 모델을 단독 기동 후 실행.
> 양자화 백엔드(실측): NVFP4 는 vLLM 0.17.1 의 `NvFp4LinearBackend.FLASHINFER_CUTLASS` 경로(SM120 동작).

## 비교 대상 (동일 dense Transformer 계열 — 변수 통제)

| 모델 | 양자화 | 디스크 | 비고 |
|------|--------|------|------|
| Llama-3.1-8B-Instruct | FP8 | 8.5GB | 작은 모델 FP8 (기준선) |
| Llama-3.1-8B-Instruct | NVFP4 (W4A4) | 5.7GB | 동일 모델 FP4 → FP8 대비 손실 측정 |
| Mistral-Small-3.2-24B-Instruct | NVFP4 (W4A4) | 15GB | 3배 큰 모델 FP4 → 작은 FP8 능가 여부 |

> ⚠ 모두 **커뮤니티/RedHat 양자화본**이며, 운영은 출처·품질을 확정해 재스테이징한다.
> ⚠ 운영 타겟인 **70B-NVFP4(~43GB)는 5090(32GB)에 미적재** → 운영 96GB 장비에서 `70B-FP8 vs 70B-NVFP4` 직접 검증 권장.

## 1. 속도 (단일요청 디코드 throughput)

| 모델 | easy 셋 | hard 셋 |
|------|------|------|
| Llama-8B-FP8 | 149 tok/s | 151 tok/s |
| Llama-8B-NVFP4 | 158 tok/s | 162 tok/s |
| Mistral-24B-NVFP4 | 72 tok/s | 73 tok/s |

- **양자화로 인한 속도차는 작다.** 같은 8B에서 FP4 가 FP8 보다 오히려 약간 빠름(메모리 대역폭↓).
- **속도는 모델 크기에 좌우.** 24B는 8B의 ~절반.

## 2. 품질 — 쉬운 셋(easy)

코드/요약/사실성/지시 등 일반 작업 5문항: **세 모델 모두 정답.**
- 8B FP8 ↔ NVFP4 품질 **사실상 동일**(FP4 손실 무시 수준).
- 24B 가 코드 견고성(`n==2` 처리)·사실성(JVM 언어에 Java 포함) 등에서 미세 우위.

## 3. 품질 — 까다로운 셋(hard) ★핵심

함정추론·논리오류·엣지케이스·엄격포맷·다단계·한국어함정 7문항 채점:

| 문제(정답) | Llama-8B-FP8 | Llama-8B-NVFP4 | Mistral-24B-NVFP4 |
|------|:---:|:---:|:---:|
| 함정추론 (5분) | ❌ 1분 | ❌ 1분 | ✅ 5분 |
| 역산수학 ($60) | ✅ | ✅ | ✅ |
| 논리오류 (대전제 거짓) | ❌ | ❌ | ✅ |
| 엣지케이스 코딩 | △ | ✅ | ✅ |
| 엄격포맷 (펜스 금지) | ✅ | ✅ | ❌ 코드펜스 추가 |
| 다단계추론 (1명) | ❌ 6명 | ❌ 6명 | △ 방향정확·미완 |
| 한국어함정 (토끼4) | ❌ | ❌ **반복붕괴** | ✅ 토끼4 |
| **정답 (대략)** | **~2.5/7** | **~2.5/7** | **~5/7** |

## 4. 결론 (운영 의사결정)

### ① 까다로운 추론에선 **모델 크기 ≫ 양자화**
- 24B-NVFP4 가 8B(FP8/NVFP4) 를 함정추론·논리·한국어함정에서 **압도**.
- 8B 두 모델은 함정에 동일하게 실패 → **"더 큰 모델을 FP4로" > "작은 모델을 FP8로"** 가 데이터로 확인됨.
- ⇒ **운영 메인을 70B-NVFP4 로 가는 전략을 지지**한다(품질 우선이면 큰 모델이 핵심).

### ② FP4 의 미세 리스크는 **작은 모델 한정**
- **8B-NVFP4 가 긴 한국어 추론에서 반복 붕괴(degeneration)** 발생("…의 수" 무한반복).
- 동일 문제에서 **24B-NVFP4 는 안정적으로 정답** → **큰 모델일수록 FP4 안정성↑**.
- 짧은 응답(easy)에선 FP8≈FP4 였으나, **긴 복잡 생성에서 작은 모델 FP4 가 불안정**할 수 있음.
- ⇒ **자동완성·경량 서브 모델은 FP8 권장**(작고 FP4 면 긴 생성 위험). **메인 대형 모델은 NVFP4 안전**.

### ③ 트레이드오프 메모
- 8B 는 FP4 가 코드 엣지케이스에서 오히려 FP8 보다 나은 경우도 있음(케이스 의존).
- 24B 는 지시 정밀도(엄격 포맷)에서 군더더기(코드펜스) 추가 — 모델별 출력 습관 차이.
- 큰 모델은 속도가 ~절반 → 자동완성처럼 빈번·저지연 작업엔 부적합, 작은 FIM 모델로 분리.

### 운영 반영 요약
| 역할 | 권장 양자화 | 근거 |
|------|------|------|
| 메인 채팅·에이전트(70B급) | **NVFP4** 또는 FP8 | 큰 모델 FP4 안정+고품질, VRAM 절반. 품질은 96GB 장비에서 `70B-FP8 vs NVFP4` 최종 확인 |
| 서브 채팅(27B급) | **FP8** | 중간 크기 FP4 의 긴생성 안정성 미검증 → 보수적으로 FP8 |
| 자동완성(7B FIM) | **FP8** | 작은 모델 FP4 는 긴 생성 degeneration 위험 |

> 한 줄: **"대형 모델은 NVFP4 로 메모리를 아끼되, 소형 모델은 FP8 로 안정성을 지킨다."**

# 부록. 자동완성 모델 비교 — StarCoder2 vs Qwen2.5-Coder (2026-06-30)

> 목적: IDE tab 자동완성(FIM) 모델 품질 비교. StarCoder2(비중국계) 대비 Qwen2.5-Coder(중국계) 정확성 측정.
> 재현: `scripts/poc_fim_compare.py <엔드포인트> <이름> <starcoder|qwen>` — 각 모델 단독 기동 후 실행. 둘 다 FP8.
> ★정책: Qwen2.5-Coder 는 중국계(Alibaba). `CLAUDE.md` 의 "중국계 금지(보안)"에 따라 채팅/에이전트엔 사용 불가.
>   자동완성 예외(코드 컨텍스트 한정·폐쇄망·외부통신0) 적용은 **보안팀 공식 승인 필요**. 본 PoC 는 기술 품질 근거 자료다.

## 채점 (6개 코드 완성 FIM 케이스)

| 케이스 | StarCoder2-7B | Qwen2.5-Coder-7B |
|--------|:---:|:---:|
| 이진탐색 | ✅(뒤 garbage) | ✅ 깔끔 |
| median 엣지케이스 | ✅(군더더기) | ✅ |
| 재귀 팩토리얼 | ✅(hallucination) | ✅ |
| dict 컴프리헨션 | ❌ 하드코딩 `{1:1,2:4..}` | ✅ 진짜 컴프리헨션 |
| 예외처리(return None) | △ `float('nan')` | ✅ `return None` 정확 |
| 클래스 메서드 | ✅ | ✅ |
| **정확도** | **~3.5/6** | **6/6** |
| 군더더기/hallucination | 많음(stop 후 무한생성, 타 파일경로) | 적음(적절히 멈춤) |
| throughput(워밍 후) | 144 tok/s | ~155 tok/s |

## 결론
- **Qwen2.5-Coder-7B 가 StarCoder2-7B 보다 자동완성 품질 명확히 우위**(정확도 6/6 vs 3.5/6, garbage 없음). 관용구(컴프리헨션)·의도파악(return None)·간결성에서 앞섬.
- **속도는 동등**(워밍 후 ~155 vs 144 tok/s) → 품질 교체 이득이 크고 속도 손해 없음.
- **권고:** 자동완성 품질을 우선한다면 Qwen2.5-Coder 가 기술적으로 최선. 단 **중국계 예외는 보안팀 승인 전제**.
  정책 유지 시 비중국계 대안은 StarCoder2-15B(더 큰 버전)/Codestral-22B. 자동완성은 빈번 호출이라 **더 작은 Qwen2.5-Coder-3B/1.5B**(속도↑)도 검토 가치.
